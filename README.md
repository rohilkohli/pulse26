# Pulse26 🏟️⚡

### GenAI Stadium Operations & Fan Experience Platform
### FIFA World Cup 2026 · Built July 9, 2026

---

## What Is This?

Pulse26 is a two-surface GenAI platform designed for the highest-stakes window of FIFA World Cup 2026 — the knockout stage (July 9–19, 2026), when host-city venues see their sharpest crowd surges, tightest multilingual fan mix, and greatest operational pressure.

**Two connected surfaces, one backend:**

| Surface | Route | Primary Users |
|---|---|---|
| **Fan Companion** (PWA) | `/fan` | Supporters at or heading to matches |
| **Venue Ops Console** | `/ops` | Venue managers, volunteers, gate staff |

Both read from the same real-time data layer — so the alert the Ops Console raises ("Gate A queue building") and the rerouting guidance the Fan Companion gives ("Try Gate C — only 40% full") are driven by the same underlying state.

## Architecture

```
pulse26/
├── server/                     # Node.js + Express + WebSocket
│   └── src/
│       ├── domain/             # Types & interfaces (ports)
│       ├── infrastructure/     # Adapters: Digital Twin, Gemini, Venues
│       ├── services/           # Alert store
│       ├── routes/             # REST API
│       └── websocket/          # WS broadcast server
└── apps/web/                   # Vite + React + TypeScript PWA
    └── src/
        ├── presentation/       # Pages, components (fan + ops)
        ├── application/        # Hooks, state, view-models
        ├── domain/             # Types (mirror of server domain)
        └── infrastructure/     # API client adapter
```

**Key architectural principle:** Every external dependency (Gemini, Maps, telemetry) sits behind an interface in `/domain` with a concrete implementation in `/infrastructure`. The Digital Twin Simulator implements the same `ITelemetryProvider` interface a real IoT feed would use — swapping to live data is a one-file change.

## Features

### Tier 1 — Fully Implemented

**🌍 Multilingual AI Concierge**
- Streaming chat backed by Gemini 2.0 Flash
- Auto-detects the fan's language and responds natively (no separate translate step)
- Supports English, Spanish, French, Portuguese at minimum; Gemini handles 70+ languages
- Context-aware: knows venue, match phase, active crowd alerts
- Falls back to canned multilingual responses if Gemini is unavailable

**🗺️ Navigation & Wayfinding**
- City-to-venue directions (walking, transit, driving)
- Venue entrance map with occupancy status per gate
- "Beat the Queue" proactive suggestion when a gate is congested
- Wheelchair-accessible routing toggle (shows accessible entrances only)
- Rideshare one-tap deeplinks (Uber / Lyft universal URIs)
- Simulated parking lot fill levels

**📊 Live Crowd Telemetry (Ops Console)**
- 12-zone occupancy heatmap per venue, updating every 5 seconds
- Color-coded: green (low) → amber (moderate) → red (high) → pulsing red (critical)
- Per-zone queue depth, wait time, and trend (rising/stable/falling)
- Match-phase-aware crowd curve visualization

**🚨 AI-Powered Operational Alerts**
- Gemini function calling → structured `Alert` objects (never free-form text)
- Every alert shows the triggering data (occupancy %, wait time, trend duration)
- Explainer: "Why did this fire?" toggle shows the AI's reasoning
- Acknowledge / Resolve workflow for ops staff
- Auto-resolution when zone clears back below threshold
- Graceful fallback to rule-based alerts if Gemini is unavailable

### Tier 2 — Implemented

**♿ Accessibility**
- WCAG 2.1 AA: all interactive elements keyboard-navigable with visible focus indicators
- ARIA roles, labels, and live regions throughout
- High-contrast mode toggle (persisted to localStorage)
- Font size cycling (85% / 100% / 115% / 130%)
- `prefers-reduced-motion` respected at component level
- Accessible routing explicitly surfaced in navigation

**🚌 Transportation**
- Transit ETA (simulated, same interface as live Maps API)
- Parking fill levels per lot
- Rideshare deeplinks: `uber://` and `lyft://` with web fallback

### Tier 3 — Scaffolded, Not Wired

- Sustainability dashboard (CO₂ comparison transit vs. driving) — data model present
- Voice-mode concierge via Web Speech API — scaffolded

## Simulated Data Disclaimer

**All crowd telemetry is generated by the Digital Twin Simulator — no real FIFA, venue, or IoT data is used.**

This is intentional and mirrors an approach already used industry-wide: every WC26 host venue runs its own per-stadium digital twin for crowd-flow modeling and operations testing without touching the live building. Pulse26's simulator is fully transparent about this in the UI.

The simulator produces realistic patterning driven by match schedule:
- Slow rise T-90min before kickoff
- Surge at kickoff ±15min
- Stable first half (concourses drain)
- Halftime spike at concessions
- Exit surge at final whistle
- Post-match drain over 30 minutes

To replace the simulator with real sensor data: change one file — [`server/src/infrastructure/digitalTwin.ts`](server/src/infrastructure/digitalTwin.ts) — implementing the `ITelemetryProvider` interface with a real IoT adapter.

## Setup

### Prerequisites
- Node.js 18+
- A Gemini API key (free from [Google AI Studio](https://aistudio.google.com/app/apikey))
- Optionally: a Google Maps API key for live routing (app works without it using simulated routes)

### Quick Start

```bash
# 1. Clone and enter the project
cd pulse26

# 2. Set up environment variables for the server
cp server/.env.example server/.env
# Edit server/.env and add your GEMINI_API_KEY

# 3. Install server dependencies
cd server && npm install

# 4. Install frontend dependencies  
cd ../apps/web && npm install

# 5. Start the server (terminal 1)
cd ../../server && npm run dev

# 6. Start the frontend (terminal 2)
cd ../apps/web && npm run dev

# 7. Open the app
# Landing:       http://localhost:5173/
# Fan Companion: http://localhost:5173/fan
# Ops Console:   http://localhost:5173/ops
```

### Without a Gemini Key

The app runs in mock mode automatically:
- Concierge returns canned multilingual responses (one per supported language)
- Alert engine uses threshold-based rules instead of Gemini reasoning
- All telemetry simulation still works normally

## API Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/api/health` | Server health check |

## Technology Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Gemini SDK | `@google/genai` | New official SDK; old `@google/generative-ai` is deprecated |
| Alert output | Structured JSON via `responseMimeType: 'application/json'` | Typed output feeds UI directly; never parsing free-form text |
| WebSocket library | `ws` (minimal) | High-throughput telemetry push; reconnect logic in client hook |
| Realtime fallback | HTTP polling every 8s | Never hard-fails; WebSocket preferred but not required |
| i18n | `react-i18next` | Industry standard, good TypeScript support, lazy-load capable |
| Concierge multilingual | System prompt + Gemini native | Detect + respond natively in one round-trip; no translation API needed |
| Route simulation | Static step arrays | Free tier Maps API has 10K calls/month limit; only call on explicit user action |
